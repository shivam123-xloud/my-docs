# Xloud Documentation Rewriting Guidelines

## Branding
- Replace **OpenStack** → **Xloud**
- Replace **KVM** → **hypervisor**
- Consistent branding across headings, commands, references

## Doc Structure (mandatory sections)
1. Title
2. Overview
3. Architecture / Concepts (if applicable)
4. Prerequisites
5. Configuration (with GUI/CLI tabs)
6. Validation / Verification
7. Best Practices (if applicable)
8. Troubleshooting (if applicable)

## Mintlify MDX Components (use extensively)
- `Tabs` + `Tab` → GUI vs CLI switching
- `Steps` + `Step` → procedures
- `Note`, `Warning`, `Tip`, `Info`
- `CodeGroup`, `CodeBlock`
- `Accordion` → optional deep explanations
- `Card` / `CardGroup` → features/use cases
- Tables wherever useful

## Configuration Section Format
```mdx
<Tabs>
  <Tab title="GUI">
    <!-- GUI steps -->
  </Tab>
  <Tab title="CLI">
    <!-- CLI steps -->
  </Tab>
</Tabs>
```

### GUI Rules
- If GUI steps exist in source → rewrite properly
- If NOT present → create realistic GUI steps based on Xloud platform
- If GUI not applicable → mention done via **Xdeploy**, show steps

### CLI Rules
- Clean, minimal commands
- Short explanation below each command

## Writing Style
- Enterprise tone (AWS/Azure level)
- Clear, direct, concise
- No casual language, no "we"/"you guys"
- Precise technical language

## Enhancements
- Add clarity where source is vague
- Add validation steps even if missing
- Add notes/tips where helpful
- Consistent formatting throughout

## Output Quality
- Production-ready, clean MDX
- Proper spacing and hierarchy
- Visually structured, easy to scan
