## Overview

The Xloud Dashboard provides a consolidated, web-based management interface for monitoring and managing the entire virtualized environment. It delivers real-time visibility into system health, instance status, resource utilization, alarms, and operational logs — all from a single pane of glass.

<CardGroup cols={3}>
  <Card title="Environment Status" icon="gauge-high">
Real-time overview of hypervisor health, resource utilization, and service availability across all configured clouds.
  </Card>
  <Card title="Alarms & Alerts" icon="bell">
Centralized alarm management with configurable thresholds for CPU, memory, disk, and network metrics.
  </Card>
  <Card title="Instance Monitoring" icon="server">
Live instance status tracking with filtering by project, cloud, status, and tags.
  </Card>
</CardGroup>

## Architecture / Concepts

### Dashboard Structure

The Xloud Dashboard is organized into four primary navigation panels:

| Panel | Description | Access Level |
|---|---|---|
| **Project** | Project-scoped view — instances, volumes, networks, images, security, and usage for the active project | All authenticated users |
| **Admin** | System-wide administration — hypervisors, services, aggregates, flavors, system status, and all-project views | Admin role only |
| **Identity** | User, group, project, role, and domain management | Admin / Identity manager |
| **Settings** | User preferences — language, timezone, table density, password changes | All authenticated users |

## Prerequisites

- Xloud Dashboard service deployed and accessible via HTTPS
- Xloud Identity (Keystone) configured with user authentication
- Compute, Network, Block Storage, and Image services operational
- Admin role assigned for access to system-level panels
- Supported browser: Chrome, Firefox, Safari, or Edge (latest versions)

## Configuration

<Tabs>
  <Tab title="GUI">
    <Steps>
      <Step title="Log In to Xloud Dashboard">
Navigate to the Xloud Dashboard URL (e.g., `https://dashboard.xloud.local`).

Enter credentials:

- **Domain** — Authentication domain (default: `Default`)
- **Username** — Xloud user account
- **Password** — Account password

Click **Sign In**.
      </Step>
      <Step title="Access the Overview Panel">
After login, the dashboard opens to the **Project** → **Compute** → **Overview** panel.

This panel displays:

- **Usage Summary** — vCPUs, RAM, disk, and instances consumed vs. quota
- **Usage Chart** — Resource utilization over the selected time range
- **Date Range Filter** — Adjust the reporting period
      </Step>
      <Step title="Switch to Admin Panel">
Click **Admin** in the left navigation to access system-wide views (requires admin role).
      </Step>
    </Steps>
  </Tab>
  <Tab title="CLI">
Dashboard configuration is managed via the `local_settings.py` file on the Dashboard server.

    <Steps>
      <Step title="Configure Overview Date Range">
```bash
vi /etc/openstack-dashboard/local_settings.py
```

```python
# Days of usage data shown in the Overview panel
# Set to None to show from beginning of current month
OVERVIEW_DAYS_RANGE = 30
```
      </Step>
      <Step title="Configure API Display Limits">
```python
# Max objects before truncation warning
API_RESULT_LIMIT = 1000

# Objects per page in tables
API_RESULT_PAGE_SIZE = 20
```
      </Step>
      <Step title="Enable Operation Logging">
```python
# Enable user action audit trail
OPERATION_LOG_ENABLED = True

# Mask sensitive fields in logs
OPERATION_LOG_OPTIONS = {
    'mask_fields': ['password', 'token'],
    'target_methods': ['POST', 'PUT', 'DELETE'],
}
```
      </Step>
      <Step title="Configure Session Management">
```python
# Session timeout (seconds)
SESSION_TIMEOUT = 3600

# Handle concurrent sessions: 'allow' or 'disconnect'
SIMULTANEOUS_SESSIONS = 'disconnect'
```
      </Step>
      <Step title="Enable/Disable Panel Features">
```python
ANGULAR_FEATURES = {
    'images_panel': True,
    'key_pairs_panel': True,
    'flavors_panel': True,
    'domains_panel': True,
    'users_panel': True,
    'groups_panel': True,
    'roles_panel': True,
}
```
      </Step>
      <Step title="Restart Dashboard Service">
```bash
systemctl restart httpd   # RHEL/CentOS
systemctl restart apache2 # Ubuntu
```
      </Step>
    </Steps>
  </Tab>
</Tabs>

### 1. Environment Overview

<Accordion title="Project Overview (All Users)">

### 3. Alarms & Notifications

<Accordion title="Alarm Configuration">

Xloud Dashboard integrates with the Telemetry service (Aodh) for alarm management.

**Path:** Admin → Monitoring → Alarms (if Telemetry is enabled)

| Alarm Type | Description |
|---|---|
| **Threshold Alarm** | Triggers when a metric crosses a defined threshold (e.g., CPU > 90%) |
| **Composite Alarm** | Combines multiple conditions with AND/OR logic |
| **Event Alarm** | Triggers on specific system events |

**Configurable metrics:**

| Metric | Description | Example Threshold |
|---|---|---|
| CPU Utilization | Per-instance or per-host CPU usage | > 90% for 5 minutes |
| Memory Usage | RAM consumption | > 85% for 10 minutes |
| Disk I/O | Read/write operations per second | > 5000 IOPS sustained |
| Network Traffic | Ingress/egress bandwidth | > 1 Gbps for 5 minutes |
| Instance Count | Number of active instances | > 95% of quota |
| Service Health | Service heartbeat status | State = Down |

<Tabs>
  <Tab title="GUI">
    <Steps>
      <Step title="Create an Alarm">
Navigate to **Admin** → **Monitoring** → **Alarms** → Click **Create Alarm**.
      </Step>
      <Step title="Configure Alarm">

- **Name** — Descriptive alarm name
- **Type** — Threshold, Composite, or Event
- **Metric** — Select the metric to monitor
- **Threshold** — Set the trigger value
- **Period** — Evaluation window (seconds)
- **Severity** — Low, Moderate, or Critical
- **Action** — Notification URL or webhook
      </Step>
      <Step title="Save">
Click **Create Alarm**. The alarm appears in the alarm list with its current state (OK, Alarm, Insufficient Data).
      </Step>
    </Steps>
  </Tab>
  <Tab title="CLI">
```bash
# Create CPU threshold alarm
xloud alarm create \
  --name high-cpu-alert \
  --type threshold \
  --meter-name cpu_util \
  --threshold 90 \
  --comparison-operator gt \
  --statistic avg \
  --period 300 \
  --evaluation-periods 3 \
  --severity critical \
  --alarm-action "http://alerts.xloud.local/webhook"

# List alarms
xloud alarm list

# Show alarm state
xloud alarm show <alarm-id>

# Delete alarm
xloud alarm delete <alarm-id>
```
  </Tab>
</Tabs>

</Accordion>

### 4. Log History & Audit Trail

<Accordion title="Operation Log">

When `OPERATION_LOG_ENABLED` is set to `True`, the Dashboard records all user actions as an audit trail.

**Captured information:**

| Field | Description |
|---|---|
| **Timestamp** | When the action occurred |
| **User** | Authenticated user who performed the action |
| **Project** | Project context |
| **Action** | HTTP method (POST, PUT, DELETE) |
| **Resource** | Target resource type and ID |
| **Result** | Success or failure status |

<Tabs>
  <Tab title="GUI">
Navigate to **Admin** → **System** → **Operation Log** to view the audit trail.

Use filters to narrow by:

- Date range
- User
- Action type
- Resource type
  </Tab>
  <Tab title="CLI">
Dashboard operation logs are stored in the configured log directory:
```bash
tail -f /var/log/horizon/operation.log
```

For Compute instance action history:
```bash
xloud server event list <server-id>
xloud server event show <server-id> <request-id>
```

For system-wide audit (via Telemetry):
```bash
xloud event list --query "event_type=compute.instance.*"
```
  </Tab>
</Tabs>

</Accordion>

<Accordion title="Instance Action History">

**Path:** Project → Compute → Instances → Select Instance → **Action Log** tab

Displays a chronological record of all actions performed on the instance:

| Field | Description |
|---|---|
| **Action** | Operation type (create, stop, start, reboot, resize, migrate, etc.) |
| **Start Time** | When the action was initiated |
| **User** | User who triggered the action |
| **Message** | Status message or error details |
| **Request ID** | Unique identifier for tracing |

</Accordion>

## Best Practices

<CardGroup cols={2}>
  <Card title="Enable Operation Logging" icon="clock-rotate-left">

  </Card>
  <Card title="Set Reasonable Page Sizes" icon="table">

  </Card>
  <Card title="Configure Alarms" icon="bell">
Set threshold alarms for critical metrics (CPU > 90%, RAM > 85%, disk > 80%). Configure webhook actions for automated incident response.
  </Card>

## Troubleshooting

<Accordion title="Dashboard Login Fails">
  **Symptom:** Unable to authenticate on the login page.

  **Resolution:**
  1. Verify Xloud Identity (Keystone) service is running: `systemctl status keystone`
  2. Check the Dashboard Keystone endpoint in `local_settings.py`:
```python
OPENSTACK_KEYSTONE_URL = "https://identity.xloud.local:5000/v3"
```
  3. Verify the user exists and has valid credentials
  4. Check Dashboard logs: `tail -f /var/log/horizon/horizon.log`
</Accordion>

<Accordion title="Admin Panels Not Visible">
  **Symptom:** Only Project and Settings panels appear; Admin panel is missing.

  **Resolution:**
  1. Verify the user has the `admin` role: `xloud role assignment list --user <user> --names`
  2. Check that the admin panel is enabled in `local_settings.py`
  3. Clear browser cache and re-login
</Accordion>

<Accordion title="Overview Shows No Usage Data">
  **Symptom:** Usage charts and metrics show zero or are empty.

  **Resolution:**
  1. Check `OVERVIEW_DAYS_RANGE` setting — a value of `1` shows only the current day
  2. Set to `None` to show data from the beginning of the month:
```python
OVERVIEW_DAYS_RANGE = None
```
  3. Verify Compute service is reporting usage: `xloud usage show --project <project>`
  4. Restart the Dashboard service after configuration changes
</Accordion>

<Accordion title="Service Shows Down in System Status">
  **Symptom:** A compute or network service shows **State: Down** in the admin panel.

  **Resolution:**
  1. Identify the affected host and service
  2. SSH to the host and check the service: `systemctl status nova-compute`
  3. Review service logs: `journalctl -u nova-compute --since "1 hour ago"`
  4. Verify message queue connectivity (RabbitMQ)
  5. Restart the service if needed: `systemctl restart nova-compute`
</Accordion>

<Accordion title="Dashboard Performance Degradation">
  **Symptom:** Pages load slowly, especially instance and volume lists.

  **Resolution:**
  1. Reduce `API_RESULT_LIMIT` and `API_RESULT_PAGE_SIZE`:
```python
API_RESULT_LIMIT = 500
API_RESULT_PAGE_SIZE = 10
```
  2. Enable caching in `local_settings.py`
  3. Check the Keystone and Nova API response times
  4. Review Apache/Nginx access logs for slow requests
  5. Consider deploying multiple Dashboard instances behind a load balancer
</Accordion>
