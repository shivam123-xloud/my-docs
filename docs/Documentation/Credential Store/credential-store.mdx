## Overview

The Xloud Credential Store (Key Manager service) provides secure storage, provisioning, and management of sensitive credential data. It supports both **internal** and **external** credential store backends for securely managing:

- Usernames and passwords (passphrases)
- Access keys and secret keys
- SSH keypairs (public and private keys)
- X.509 certificates and certificate chains
- Symmetric encryption keys
- Raw binary secrets

  <Card title="Internal Store" icon="database">
Built-in encrypted storage using software-based cryptographic plugins. Suitable for standard deployments.
  </Card>
  <Card title="External Store" icon="vault">
Integration with external key management systems — HSMs, KMIP servers, HashiCorp Vault, and Dogtag CA.

## Architecture / Concepts

### Supported Secret Types

The Credential Store supports six distinct secret types:

| Secret Type | Description | Use Case |
|---|---|---|
| **symmetric** | Byte arrays for symmetric encryption keys | AES-256 keys, HMAC keys |
| **public** | Public key of an asymmetric keypair | RSA/EC public keys, SSH public keys |
| **private** | Private key of an asymmetric keypair | RSA/EC private keys, SSH private keys |
| **passphrase** | Plain text passphrases and passwords | Service account passwords, access/secret keys |
| **certificate** | X.509 certificates and certificate chains | TLS/SSL certificates, SSH certificates |
| **opaque** | Raw binary data | Any arbitrary secret data |

### Container Types

Secrets can be logically grouped using containers:

| Container Type | Contents | Use Case |
|---|---|---|
| **Generic** | Unlimited secret references with custom names | Group related credentials |
| **RSA** | `public_key`, `private_key`, `private_key_passphrase` | Store complete RSA keypairs |
| **Certificate** | `certificate`, `private_key`, `private_key_passphrase`, `intermediates` | Store TLS certificate chains |

### Internal vs External Backends

<Tabs>
  <Tab title="Internal Backends">
Internal backends store encrypted secrets within the Xloud infrastructure:

| Backend | Description | Security Level |
|---|---|---|
| **Simple Crypto** | Software-based AES encryption using a master key stored in configuration | Standard — suitable for development and non-critical workloads |
| **PKCS#11** | Hardware Security Module (HSM) integration via PKCS#11 interface | High — master key stored in tamper-resistant hardware |

    <Note>
Simple Crypto stores the master encryption key (KEK) in `barbican.conf`. For production deployments, use PKCS#11 with an HSM or an external backend.
    </Note>
  </Tab>
  <Tab title="External Backends">
External backends delegate secret storage to specialized external systems:

| Backend | Description | Security Level |
|---|---|---|
| **KMIP** | Key Management Interoperability Protocol server integration | Enterprise — OASIS standard compliance |
| **HashiCorp Vault** | Integration with Vault for centralized secret management | Enterprise — dynamic secrets, audit logging |
| **Dogtag** | Red Hat Certificate System / Dogtag PKI integration | Enterprise — certificate authority with KRA |
| **Castellan** | Abstraction layer supporting pluggable key manager backends | Varies — depends on underlying backend |

    <Tip>
External backends are recommended for production deployments where compliance requirements mandate dedicated key management infrastructure.
    </Tip>
  </Tab>
</Tabs>

### Credential Flow

```
Application/VM → Xloud API → Credential Store (Barbican) → Backend (Internal or External)
                                    ↓
                              Access Control (RBAC + ACL)
                                    ↓
                              Encrypted Storage / HSM / Vault
```

## Prerequisites

- Xloud Identity (Keystone) configured with service endpoints for the Key Manager
- Database backend for the Credential Store service (MySQL/MariaDB or PostgreSQL)
- Message queue access (RabbitMQ)
- For PKCS#11: HSM device accessible from the Credential Store host
- For KMIP: KMIP-compliant server reachable over the network
- For HashiCorp Vault: Vault server deployed with AppRole or Token authentication
- For Dogtag: Dogtag PKI server with KRA (Key Recovery Authority) configured

## Configuration

<Tabs>
  <Tab title="GUI">
    <Steps>
      <Step title="Access Credential Store Settings">
Log in to the **Xloud Dashboard** → Navigate to **Admin** → **Key Manager** → **Secret Store Configuration**.
      </Step>
      <Step title="View Active Backend">
The dashboard displays the currently active secret store backend and its status. If multi-store is enabled, each project's preferred store is shown.
      </Step>
      <Step title="Store a Credential">
Navigate to **Project** → **Key Manager** → **Secrets** → Click **Create Secret**.

Fill in:

- **Name** — Descriptive name for the credential
- **Secret Type** — Select from: symmetric, public, private, passphrase, certificate, opaque
- **Payload** — Enter the secret value
- **Content Type** — `text/plain` for passphrases, `application/octet-stream` for binary
- **Expiration** — Optional expiry date
- Click **Create**
      </Step>
      <Step title="Store a Keypair (RSA Container)">
Navigate to **Key Manager** → **Containers** → Click **Create Container**.

- **Type** — Select **RSA**
- Add secret references for `public_key`, `private_key`, and optionally `private_key_passphrase`
- Click **Create**
      </Step>
      <Step title="Store a Certificate">
Navigate to **Key Manager** → **Containers** → Click **Create Container**.

- **Type** — Select **Certificate**
- Add references for `certificate`, `private_key`, and optionally `intermediates`
- Click **Create**
      </Step>
    </Steps>

    <Note>
If the GUI is not available, use **Xdeploy** to configure the Credential Store backend on the Key Manager nodes.
    </Note>
  </Tab>

### Backend Configuration

    <Steps>
      <Step title="Configure Internal Backend (Simple Crypto)">
Edit the Credential Store configuration:
```bash
vi /etc/barbican/barbican.conf
```

```ini
[secretstore]
enabled_secretstore_plugins = store_crypto

[crypto]
enabled_crypto_plugins = simple_crypto

[simple_crypto_plugin]
kek = dGhpcyBpcyBhIHRlc3Qga2V5IGZvciBiYXJiaWNhbg==
```

        <Warning>
The `kek` (Key Encryption Key) is stored in plain text in the configuration file. Restrict file permissions to `600` and use PKCS#11 or an external backend for production.
        </Warning>
      </Step>

      <Step title="Configure Internal Backend (PKCS#11 / HSM)">
```ini
[secretstore]
enabled_secretstore_plugins = store_crypto

[crypto]
enabled_crypto_plugins = p11_crypto

[p11_crypto_plugin]
library_path = /usr/lib/libCryptoki2_64.so
login = <hsm-pin>
mkek_label = xloud_master_kek
mkek_length = 32
hmac_label = xloud_hmac_key
```

Generate the master key on the HSM:
```bash
barbican-manage hsm pkek_rewrap --library-path /usr/lib/libCryptoki2_64.so \
  --slot-id 1 --login <hsm-pin> --label xloud_master_kek
```
      </Step>

      <Step title="Configure External Backend (KMIP)">
```ini
[secretstore]
enabled_secretstore_plugins = kmip_plugin

[kmip_plugin]
username = xloud-keymanager
password = <kmip-password>
host = kmip-server.xloud.local
port = 5696
ssl_ca_certs = /etc/barbican/certs/kmip-ca.pem
ssl_cert_file = /etc/barbican/certs/kmip-client.pem
ssl_key_file = /etc/barbican/certs/kmip-client-key.pem
```
      </Step>

      <Step title="Configure External Backend (HashiCorp Vault)">
```ini
[secretstore]
enabled_secretstore_plugins = vault_plugin

[vault_plugin]
root_token_id = <vault-token>
vault_url = https://vault.xloud.local:8200
use_ssl = true
ssl_ca_certs_file = /etc/barbican/certs/vault-ca.pem
```
      </Step>

      <Step title="Configure External Backend (Dogtag)">
```ini
[secretstore]
enabled_secretstore_plugins = dogtag_crypto

[dogtag_plugin]
pem_path = /etc/barbican/certs/dogtag-agent.pem
dogtag_host = dogtag.xloud.local
dogtag_port = 8443
nss_db_path = /var/lib/barbican/dogtag-nss
```
      </Step>

      <Step title="Enable Multi-Store (Optional)">
Configure multiple backends simultaneously with project-level assignment:
```ini
[secretstore]
enable_multiple_secret_stores = true
stores_lookup_suffix = software, hsm, vault

[secretstore:software]
secret_store_plugin = store_crypto
crypto_plugin = simple_crypto

[secretstore:hsm]
secret_store_plugin = store_crypto
crypto_plugin = p11_crypto

[secretstore:vault]
secret_store_plugin = vault_plugin
```

Set a project's preferred store:
```bash
curl -X POST https://keymanager.xloud.local/v1/secret-stores/<store-id>/preferred \
  -H "X-Auth-Token: <admin-token>" \
  -H "X-Project-Id: <project-id>"
```
      </Step>

      <Step title="Restart the Service">
```bash
systemctl restart barbican-api barbican-worker
```
      </Step>
    </Steps>

### Credential Operations

    <Steps>
      <Step title="Store a Password / Access Key">
```bash
# Store a password
xloud secret store \
  --name "db-admin-password" \
  --secret-type passphrase \
  --payload "S3cur3P@ssw0rd" \
  --payload-content-type "text/plain"

# Store an access key
xloud secret store \
  --name "s3-access-key" \
  --secret-type passphrase \
  --payload "AKIAIOSFODNN7EXAMPLE" \
  --payload-content-type "text/plain"

# Store a secret key
xloud secret store \
  --name "s3-secret-key" \
  --secret-type passphrase \
  --payload "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY" \
  --payload-content-type "text/plain"
```
      </Step>

      <Step title="Store an SSH Keypair">
```bash
# Store SSH private key
xloud secret store \
  --name "ssh-private-key" \
  --secret-type private \
  --payload-content-type "application/octet-stream" \
  --payload-content-encoding base64 \
  --payload "$(base64 ~/.ssh/id_rsa)"

# Store SSH public key
xloud secret store \
  --name "ssh-public-key" \
  --secret-type public \
  --payload-content-type "application/octet-stream" \
  --payload-content-encoding base64 \
  --payload "$(base64 ~/.ssh/id_rsa.pub)"

# Group in RSA container
xloud secret container create \
  --name "ssh-keypair" \
  --type rsa \
  --secret "private_key=<private-key-ref>" \
  --secret "public_key=<public-key-ref>"
```
      </Step>

      <Step title="Store an X.509 / SSH Certificate">
```bash
# Store TLS certificate
xloud secret store \
  --name "tls-certificate" \
  --secret-type certificate \
  --payload-content-type "application/octet-stream" \
  --payload-content-encoding base64 \
  --payload "$(base64 server.crt)"

# Store in certificate container with private key
xloud secret container create \
  --name "tls-bundle" \
  --type certificate \
  --secret "certificate=<cert-ref>" \
  --secret "private_key=<key-ref>" \
  --secret "intermediates=<intermediate-ref>"
```
      </Step>

      <Step title="Store a Symmetric Encryption Key">
```bash
xloud secret store \
  --name "aes-256-key" \
  --secret-type symmetric \
  --algorithm aes \
  --bit-length 256 \
  --mode cbc \
  --payload-content-type "application/octet-stream" \
  --payload-content-encoding base64 \
  --payload "$(openssl rand -base64 32)"
```
      </Step>

      <Step title="Retrieve a Credential">
```bash
# Get secret metadata
xloud secret get <secret-ref>

# Get secret payload
xloud secret get <secret-ref> --payload

# Get payload to file
xloud secret get <secret-ref> --payload --file output.key
```
      </Step>

      <Step title="Delete a Credential">
```bash
xloud secret delete <secret-ref>
```
      </Step>

      <Step title="List and Search Credentials">
```bash
# List all secrets
xloud secret list

# Filter by name
xloud secret list --name "db-*"

# Filter by type
xloud secret list --secret-type passphrase

# Filter by algorithm
xloud secret list --algorithm aes --bit-length 256
```
      </Step>
    </Steps>
  </Tab>
</Tabs>

## API Reference

### Secrets API

| Operation | Method | Endpoint | Description |
|---|---|---|---|
| List secrets | `GET` | `/v1/secrets` | List all secrets with filtering and pagination |
| Create secret | `POST` | `/v1/secrets` | Create a new secret (with or without payload) |
| Get secret metadata | `GET` | `/v1/secrets/{id}` | Retrieve secret metadata |
| Get secret payload | `GET` | `/v1/secrets/{id}/payload` | Retrieve the actual secret data |
| Update secret payload | `PUT` | `/v1/secrets/{id}` | Add payload to a metadata-only secret |
| Delete secret | `DELETE` | `/v1/secrets/{id}` | Permanently delete a secret |

**Supported query filters for listing:**

| Filter | Description |
|---|---|
| `name` | Filter by secret name |
| `secret_type` | Filter by type (symmetric, public, private, passphrase, certificate, opaque) |
| `alg` | Filter by algorithm |
| `bits` | Filter by bit length |
| `mode` | Filter by mode |
| `created` / `updated` | Date filters with operators (gt, gte, lt, lte) |
| `expiration` | Filter by expiration date |
| `sort` | Sort by created, expiration, name, status, updated |
| `offset` / `limit` | Pagination (max 100 per page) |

### Containers API

| Operation | Method | Endpoint | Description |
|---|---|---|---|
| List containers | `GET` | `/v1/containers` | List all containers |
| Create container | `POST` | `/v1/containers` | Create a new container (generic, RSA, or certificate) |
| Get container | `GET` | `/v1/containers/{id}` | Retrieve container details and secret references |
| Delete container | `DELETE` | `/v1/containers/{id}` | Delete a container |
| Add secret to container | `POST` | `/v1/containers/{id}/secrets` | Add a secret reference (generic containers only) |
| Remove secret from container | `DELETE` | `/v1/containers/{id}/secrets` | Remove a secret reference (generic containers only) |

### Secret Stores API (Multi-Store)

| Operation | Method | Endpoint | Description |
|---|---|---|---|
| List secret stores | `GET` | `/v1/secret-stores` | List all configured backends |
| Get secret store | `GET` | `/v1/secret-stores/{id}` | Show backend details |
| Get project store | `GET` | `/v1/secret-stores/preferred` | Show project's preferred store |
| Set project store | `POST` | `/v1/secret-stores/{id}/preferred` | Set preferred store for a project |
| Remove project store | `DELETE` | `/v1/secret-stores/{id}/preferred` | Remove preferred store assignment |

## Credential Type Mapping

| Credential Type | Secret Type | Container Type | Storage Example |
|---|---|---|---|
| Username / Password | `passphrase` | `generic` | Service account credentials |
| Access Key | `passphrase` | `generic` | S3-compatible access keys |
| Secret Key | `passphrase` | `generic` | S3-compatible secret keys |
| SSH Private Key | `private` | `rsa` | SSH authentication keys |
| SSH Public Key | `public` | `rsa` | SSH authentication keys |
| SSH Certificate | `certificate` | `certificate` | SSH CA-signed certificates |
| TLS/SSL Certificate | `certificate` | `certificate` | HTTPS certificates |
| TLS Private Key | `private` | `certificate` | HTTPS private keys |
| Symmetric Key | `symmetric` | `generic` | Encryption keys (AES, HMAC) |
| API Token | `opaque` | `generic` | Bearer tokens, API keys |

## Backend Comparison

| Feature | Simple Crypto | PKCS#11 (HSM) | KMIP | HashiCorp Vault | Dogtag |
|---|---|---|---|---|---|
| **Type** | Internal | Internal | External | External | External |
| **Key Storage** | Config file | Hardware device | KMIP server | Vault server | Dogtag KRA |
| **FIPS 140-2** | No | Yes (HSM dependent) | Yes (server dependent) | Enterprise only | Yes |
| **Multi-Tenancy** | Yes | Yes | Yes | Yes | Yes |
| **Certificate Authority** | No | No | No | Yes (PKI engine) | Yes |
| **Dynamic Secrets** | No | No | No | Yes | No |
| **Audit Logging** | Via Xloud | Via HSM | Via KMIP server | Built-in | Built-in |
| **Recommended For** | Dev/Test | Production (high security) | Enterprise compliance | Production (flexibility) | PKI-centric deployments |

## Best Practices

<CardGroup cols={2}>
  <Card title="Use External Backends in Production" icon="vault">
Deploy PKCS#11 (HSM), KMIP, or HashiCorp Vault for production workloads. Simple Crypto is acceptable only for development and testing.
  </Card>
  <Card title="Set Expiration Dates" icon="calendar-xmark">
Always set expiration dates on secrets, especially passwords and API keys. This enforces credential rotation and limits exposure from stale secrets.
  </Card>
  <Card title="Use Containers for Keypairs" icon="box">
Group related secrets (public key + private key + passphrase) in RSA or Certificate containers rather than storing them as individual secrets.
  </Card>
  <Card title="Restrict Access with ACLs" icon="lock">
Apply per-secret ACLs to limit which users and services can read, update, or delete specific credentials. Default project-level access may be too broad.
  </Card>
  <Card title="Rotate Credentials Regularly" icon="rotate">
Implement automated credential rotation. Create new secrets, update consuming services, then delete old secrets.
  </Card>
  <Card title="Secure the KEK" icon="key">
For Simple Crypto backend, restrict `/etc/barbican/barbican.conf` to `root:barbican` with `640` permissions. Migrate to HSM for production.
  </Card>
</CardGroup>

<Warning>
  Never store the Key Encryption Key (KEK) in version control, shared storage, or unencrypted backups. The KEK protects all secrets in the Credential Store — its compromise exposes every stored credential.
</Warning>

## Troubleshooting

<Accordion title="Secret Creation Fails with 403 Forbidden">
  **Symptom:** `HTTP 403` when creating a secret.

  **Resolution:**
  1. Verify the user has the `creator` role for the Key Manager service
  2. Check project quotas — secret count or total payload size may be exceeded
  3. Review Barbican policy file: `/etc/barbican/policy.yaml`
  4. Confirm the Keystone endpoint is correctly registered
</Accordion>

<Accordion title="Cannot Retrieve Secret Payload">
  **Symptom:** Secret metadata is returned but payload retrieval fails with `406 Not Acceptable`.

  **Resolution:**
  1. Set the correct `Accept` header matching the secret's content type
  2. Use the `/v1/secrets/{id}/payload` endpoint (not the metadata endpoint)
  3. Verify the secret has a payload — metadata-only secrets return no payload
  4. Check ACLs on the specific secret
</Accordion>

<Accordion title="HSM / PKCS#11 Connection Fails">
  **Symptom:** Barbican fails to start or secrets fail with backend errors.

  **Resolution:**
  1. Verify the HSM library path: `ls -la /usr/lib/libCryptoki2_64.so`
  2. Test HSM connectivity: `pkcs11-tool --module <library-path> --list-slots`
  3. Verify the PIN/login credentials in `barbican.conf`
  4. Check that the master KEK label exists on the HSM
  5. Review Barbican logs: `journalctl -u barbican-api`
</Accordion>

<Accordion title="Vault Backend Returns Authentication Errors">
  **Symptom:** Secrets fail with Vault authentication errors.

  **Resolution:**
  1. Verify the Vault token is valid: `vault token lookup`
  2. Check the Vault URL and SSL certificate paths in `barbican.conf`
  3. Ensure the Vault policy grants read/write access to the Barbican path
  4. Rotate the Vault token if expired and update `barbican.conf`
  5. Restart: `systemctl restart barbican-api barbican-worker`
</Accordion>

<Accordion title="Multi-Store Project Assignment Not Working">
  **Symptom:** Secrets are stored in the default backend instead of the project's preferred store.

  **Resolution:**
  1. Verify multi-store is enabled: check `enable_multiple_secret_stores = true` in `barbican.conf`
  2. List configured stores: `GET /v1/secret-stores`
  3. Verify the project's preferred store: `GET /v1/secret-stores/preferred`
  4. Re-assign if needed:
```bash
curl -X POST https://keymanager.xloud.local/v1/secret-stores/<store-id>/preferred \
  -H "X-Auth-Token: <token>" -H "X-Project-Id: <project-id>"
```
</Accordion>
